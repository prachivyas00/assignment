# Assumptions Log

Every decision you make about ambiguous data, unclear requirements, or edge case handling should be logged here. This document is as important as your code.
If a number on the dashboard can't be traced back to a documented decision, it shouldn't be on the dashboard.

## How to Use This Template

For each assumption:
1. State the assumption clearly
2. Explain your rationale (what evidence led you here?)
3. Assess the impact if the assumption turns out to be wrong
4. Date it so we know the sequence of decisions

## Data Interpretation

| # | Assumption | Rationale | Impact if Wrong | Date |
|---|-----------|-----------|----------------|------|
| 1 | Subscription activity is determined by the `status` field, not by `current_term_start` or `current_term_end` dates | When I first ran the ARR calculation it returned zero. I added a debug log and found that all term dates in the dataset fall in 2024-2025, which predates today (April 2026). Filtering by term dates would silently exclude every single subscription. The `status` field, "active", "cancelled", "in_trial" etc. is what actually reflects the real state of a subscription, so I used that instead | ARR would return $0 if term dates were used as the primary filter. This would be the wrong number to present to the board | 12-04-2026 |
| 2 | MRR is calculated from `plan.price` divided by the billing interval, because the `mrr` field is null in the raw Chargebee JSON | I inspected the raw `chargebee_subscriptions.json` file directly and found that the `mrr` field was null for every single record. The `plan.price` field was always populated and represents what the customer is billed. For monthly plans MRR = plan.price. For annual plans MRR = plan.price / 12. This is standard SaaS revenue recognition | If plan.price doesn't reflect what the customer is actually charged after discounts or credits, MRR and ARR would be overstated | 13-04-2026 |
| 3 | The raw Chargebee JSON uses completely different field names from the TypeScript types defined in the codebase | I discovered this by printing a raw subscription record to the terminal. The raw data has `id` where the type expects `subscription_id`, `plan.name` where it expects `plan.plan_name`, `plan.interval` where it expects `billing_period_unit`, and `customer.id` where it expects `customer.customer_id`. The TypeScript types look like they were written as an idealized model, not derived from the actual API response. I wrote the ingestion loader to explicitly map raw field names to the expected type structure | If the raw schema changes in a future Chargebee export, the loader would silently break and return empty or malformed data | 12-04-2026 |
| 4 | Stripe payment amounts in the CSV are in dollars (e.g. 145.16), not cents | Looking at the raw values in `stripe_payments.csv`, amounts had decimal points consistent with dollar values. I multiplied by 100 on ingestion to normalize everything to cents, which is how Chargebee stores amounts and how all internal calculations are done. Working in whole numbers avoids floating point issues with money | If amounts were already in cents, every Stripe revenue figure would be overstated by 100x | 12-04-2026 |
| 5 | Legacy invoice dates where the first number exceeds 12 are treated as DD/MM/YYYY. Everything else defaults to MM/DD/YYYY | The legacy system was used by operators in different countries. A date like "25/03/2023" can only be DD/MM/YYYY because there is no 25th month. For genuinely ambiguous dates like "03/04/2023" I defaulted to MM/DD/YYYY (US format) because the company that built the legacy system was US-based. I checked a sample of surrounding dates in the file and US format was consistent with the sequence | Up to ~5% of invoices could have the wrong month assigned if the default is wrong. This would affect monthly revenue attribution but not annual totals | 12-04-2026 |
| 6 | A `payment_ref` value in legacy invoices is only treated as a Stripe charge ID if it starts with `ch_` or `pi_`. Everything else is discarded | The legacy system used this field inconsistently, sometimes it contains a real Stripe charge ID, sometimes it contains an internal reference number, sometimes it's null. Stripe charge IDs always begin with `ch_` or `pi_` followed by alphanumeric characters. Anything else is not a valid Stripe reference and trying to cross-reference it would produce false matches | We lose the ability to link legacy invoices that have internal reference numbers instead of Stripe IDs. Those invoices remain as orphan records | 13-04-2026 |
| 7 | FX conversion uses the exchange rate from the payment date, not today's rate | If I used today's rate, the USD value of a historical EUR payment would change every single day as exchange rates move. That makes the numbers non-auditable, the CFO would see different revenue figures every time they refresh. Using the payment-date rate means the number is fixed and traceable. I implemented a 5-day lookback window to handle weekends and public holidays where rate data doesn't exist | If the wrong date's rate is used, non-USD revenue figures could be off by 1-3% depending on how much the currency moved | 12-04-2026 |


## Metric Definitions

| # | Assumption | Rationale | Impact if Wrong | Date |
|---|-----------|-----------|----------------|------|
| 1 | ARR = sum of MRR across all active subscriptions × 12, regardless of whether a subscription is billed monthly or annually | This is the standard SaaS definition. A customer on a $100/month plan and a customer on a $1,200/year plan both contribute $1,200 ARR. Normalizing to monthly and annualizing makes them comparable | None. This is the universal definition. Any deviation from it would make our numbers incomparable to industry benchmarks | 12-04-2026 |
| 2 | Subscriptions in `in_trial` status are excluded from ARR by default | A trial customer hasn't paid yet and may never convert. Including them would inflate ARR and make it indefensible to the board. The CFO brief specifically asked for "clean, defensible numbers" | ARR would be overstated if trials are included. Worse, if those trials churn at conversion the ARR would drop suddenly with no real business change to explain it | 13-04-2026 |
| 3 | Annual subscriptions are attributed at 1/12 of their total price per month for MRR purposes | A customer who pays $1,200 upfront for a year doesn't generate $1,200 of MRR in January and $0 for the rest of the year. Revenue recognition spreads it evenly. This also prevents MRR from spiking artificially in renewal months | If annual payments were counted as lump sums, MRR would be wildly volatile month to month with no real business explanation | 13-04-2026 |
| 4 | Customer health score weights: subscription status 50%, plan tier 30%, MRR 20% | Status is the single strongest signal of churn risk, a non-renewing or paused subscription is a clear warning sign regardless of plan. Plan tier reflects how deeply embedded the customer is in the product (Enterprise customers tend to have more integrations and stakeholders, making churn harder). MRR reflects financial commitment. I weighted status heaviest because it's the most direct signal | If the weights are wrong, individual customer scores would be miscalibrated. But the relative ranking, who's healthiest vs who's at risk, would still be directionally correct | 13-04-2026 |


## Business Logic

| # | Assumption | Rationale | Impact if Wrong | Date |
|---|-----------|-----------|----------------|------|
| 1 | A customer with active subscriptions in both Stripe and Chargebee with overlapping dates is a `true_duplicate` | A proper billing migration should cancel the old system's subscription before or immediately after activating the new one. If both are active at the same time, the migration wasn't done cleanly and revenue is being double-counted. I reviewed the data and found no cases where a customer legitimately needed active subscriptions in both systems simultaneously | We could misclassify a customer who genuinely uses different products billed through different systems. I found no such cases in this dataset but it's worth a human review pass | 13-04-2026 |
| 2 | Sequential subscriptions where Stripe ends and Chargebee starts within a 30-day gap are classified as `migration` rather than `true_duplicate` | 30 days covers a standard monthly billing cycle. If a customer's last Stripe payment was in December and their Chargebee subscription started in mid-January, that's a planned migration, not a lapse in service or a data error. I didn't want to flag clean migrations as problems because that would create noise in the discrepancy report | A gap longer than 30 days might actually be a genuine service lapse rather than a migration. Those cases would be incorrectly classified as `migration` when they should be `uncertain` | 13-04-2026 |
| 3 | The confidence threshold for duplicate detection is 0.5, lower than the 0.6 default I used for general entity matching | Company names vary significantly between billing systems. "Acme Corp" in Stripe and "ACME Corporation Ltd." in Chargebee is obviously the same company but scores lower than you'd expect if you compare raw strings. I lowered the threshold because the cost of missing a true duplicate (ARR gets overstated, CFO presents wrong numbers to the board) is much higher than the cost of a false positive (a record gets flagged that needs a quick human review) | A lower threshold means more false positives, records that get flagged as potential duplicates but aren't. These would need human review to dismiss | 12-04-2026 |
| 4 | Stripe subscriptions are reconstructed from payment history by grouping all payments sharing a `subscription_id` and treating the range from first to last payment plus one month as the active window | The Stripe payments CSV doesn't include subscription start and end dates, only individual payment records. Grouping by subscription_id and inferring the window from payments is the most reliable reconstruction available with this data | Could underestimate the subscription window if some payments failed and were never retried, or overestimate if a subscription renewed late | 12-04-2026 |


## Exclusions & Edge Cases

| # | Assumption | Rationale | Impact if Wrong | Date |
|---|-----------|-----------|----------------|------|
| 1 | Cancelled and paused subscriptions are excluded from all ARR and health calculations | Cancelled subscriptions generate $0 recurring revenue. Paused subscriptions are temporarily inactive, including them would overstate ARR | If a paused subscription is about to resume, excluding it slightly understates near-term ARR. This is conservative and defensible | 13-04-2026 |
| 2 | Stripe payments with a `refund_id` or `dispute_id` are excluded from actual revenue totals | A refunded payment means the money was returned to the customer. A disputed payment is contested and may be reversed. Neither represents collected revenue | If a dispute is later resolved in the company's favor, the revenue would be understated until the next reconciliation run | 13-04-2026 |
| 3 | Customers with no billing address country in Chargebee are grouped under "Unknown" in regional breakdowns | The raw Chargebee JSON does not include a billing address at the customer level for any record in this dataset. I couldn't derive country from the data available without joining to Salesforce, which I haven't fully implemented yet | The regional breakdown is currently not useful, all 750 customers appear under "Unknown". This is a known limitation I've documented in the architecture doc | 13-04-2026 |
| 4 | Legacy invoices with null or non-Stripe payment_ref values are ingested but not cross-referenced | The majority of legacy invoices have null payment_ref. Trying to match them to Stripe payments without a reference key would require pure name matching, which is too imprecise for revenue reconciliation (the cost of a false match is incorrectly attributing revenue) | These orphan invoices are counted in ingestion totals but don't contribute to cross-system reconciliation. Their revenue is not reflected in the discrepancy analysis | 13-04-2026 |
